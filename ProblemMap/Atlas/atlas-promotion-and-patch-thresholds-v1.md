# Atlas Promotion and Patch Thresholds 🏗️

## Problem Map 3.0 Troubleshooting Atlas
> Quick links, growth discipline, and beginner-friendly rules for what has earned promotion and what should stay patch-sensitive

This page exists because Atlas is now large enough that growth itself needs structure.

Earlier pages already clarified:

- the family layer
- the boundary layer
- the subtree layer
- the fit layer
- the first-fix layer
- the evidence layer
- the output-contract layer
- the overlay layer

That is strong progress.

But once public decomposition grows, a new problem appears:

how do we decide whether a structure has actually earned stronger status?

Without a page like this, several forms of drift become likely.

A system may start doing things like:

- promoting a useful phrase into a stable structure too early
- treating a repeated example like a durable branch
- mistaking more pages for more truth
- mistaking a good public explanation for a frozen concept
- treating patch work as if it proves instability
- treating popularity, convenience, or elegance as if they were structural validation

This page exists to stop that.

It gives Atlas a public growth discipline for two linked questions:

- what has earned stronger promotion
- what should remain patch-sensitive

This is not bureaucracy for its own sake.

It is what keeps a system from mutating by enthusiasm alone.

## Why this page exists

A mature system does not only need definitions.

It also needs thresholds.

That matters because not every meaningful idea deserves the same public status.

Some things are:

- useful references
- helpful teaching patterns
- stable enough public pages
- patch-sensitive expansions
- support-layer structures
- only very rarely, candidates for stronger frozen status

If these levels are not distinguished, growth becomes noisy.

Then the system slowly starts to confuse:

- public usefulness with structural stability
- repeated mention with earned promotion
- clean wording with real boundary stability
- worked examples with generalizable structure
- successful demos with core eligibility

So this page exists to make that ladder explicit.

Promotion should be earned.  
Patch status should be explicit.  
Neither one should be rhetorical theater.

## How to use this page

Use this page after the structure itself has already been made visible.

### Step 1
Clarify what kind of thing you are evaluating.

Ask:
- is this a wording
- a repeated pattern
- a teaching surface
- a subtree candidate
- a fit phrase
- an overlay pattern
- a full public page
- a support-layer concept

### Step 2
Ask what current level it already deserves.

Possible answers include:
- referenced only
- teaching support
- stable enough public page
- frozen support
- frozen core eligible

### Step 3
Ask whether the current evidence, boundary stability, and wording stability actually support upward movement.

### Step 4
If not, keep the item patch-sensitive or lower-status rather than inflating it.

That order matters.

Promotion is not the starting point.  
It is the result of enough structural support.

---

## What promotion means

Promotion means that a wording, branch, page, or structure has earned a stronger public stability status than it had before.

In Atlas terms, promotion should usually mean at least one of these things:

- the structure now has clearer routing value
- the structure now has clearer boundary separation
- the structure now has more stable wording
- the structure now has stronger public readability without losing discipline
- the structure now supports safer downstream use

Promotion is not just “I like this idea.”

Promotion is a public-status change.

That is why it needs thresholds.

---

## What patch-sensitive means

Patch-sensitive means the item is real enough to work on, but not yet stable enough to be treated as stronger public structure without caution.

That does **not** mean the item is useless.

It often means:

- the wording is still moving
- the neighboring boundaries are still not clean enough
- the evidence is still thinner than the public confidence might imply
- the worked examples are promising but not yet enough
- the structure helps, but its exact public role is still being refined

Patch-sensitive is not a weakness label.

It is a growth label.

It says the item should remain under explicit discipline while it matures.

---

## Why promotion and patch thresholds belong together

These two topics belong together because many public mistakes happen exactly at their boundary.

A system may:

- promote something that should still be patch-sensitive
- leave something patch-sensitive forever even though it has earned stronger status
- treat support-layer stability as if it were core stability
- talk about a patch as if the core had been invalidated

So Atlas needs both ideas on the same page:

promotion  
and  
patch threshold

That keeps upward movement from becoming arbitrary.

## Promotion classes

This first public version uses a simple promotion ladder.

---

## 1. Referenced only

### Meaning
The structure is real enough to mention, but not stable enough to treat as an independent public asset.

### Typical use
- a branch is visible
- a wording pattern is useful
- a distinction may matter later
- a public page would still be premature

### What this does not mean
It does **not** mean the structure is fake.
It means the structure has not yet earned stronger independent public status.

---

## 2. Teaching support

### Meaning
The structure is useful enough for examples, interpretation, or public teaching, even if it is not yet strong enough to act like a more stable structural asset.

### Typical use
- casebook explanation
- worked examples
- public orientation material
- temporary explanatory support

### What this does not mean
Teaching usefulness is not the same thing as structural stability.

A concept can be good for teaching before it is good for promotion.

---

## 3. Stable enough public page

### Meaning
The structure has earned its own public page or explicit public slot because it provides clear value and sufficiently stable meaning.

### Typical use
- a middle-layer document
- a public control page
- a stable explanatory layer
- a structure that helps route, interpret, or constrain later work

### What this does not mean
It does **not** automatically mean frozen support or core eligibility.

It means the page is stable enough to exist publicly in its own right.

---

## 4. Frozen support

### Meaning
The structure is stable enough to be treated as a dependable support-layer asset in the public Atlas stack.

### Typical use
- support pages that hold stable language and stable function
- documents that are not the mother structure, but are trusted support surfaces

### What this does not mean
Frozen support is still not identical to frozen core.

A support layer can be highly stable without being the core body.

---

## 5. Frozen core eligible

### Meaning
The structure has become strong enough in evidence, wording stability, boundary stability, and systemic importance that it may be a candidate for stronger core-level treatment.

### Typical use
This should be rare.

### What this does not mean
This is not an emotional reward tier.  
It should never be granted because something is popular, beautiful, or rhetorically impressive.

Frozen core eligibility should be hard to earn.

---

## Promotion ladder summary table 📚

| Promotion class | What it means | Good use case | Main danger if misused |
|---|---|---|---|
| Referenced only | real enough to mention, not yet stable enough to stand alone | early pattern visibility | promoting too early |
| Teaching support | useful for examples and explanation | casebook or orientation help | mistaking teaching clarity for structural stability |
| Stable enough public page | strong enough for independent public decomposition | middle-layer clarification page | treating page existence as core legitimacy |
| Frozen support | dependable support-layer stability | trusted non-core support asset | confusing support with core |
| Frozen core eligible | rare candidate for stronger core treatment | highly stable, systemically important structure | inflation by rhetoric, popularity, or repetition |

## Patch-sensitive classes

Promotion alone is not enough.  
Atlas also needs a public language for what should remain patch-sensitive.

---

## 1. Wording patch-sensitive

### Meaning
The structure exists, but the wording is not yet stable enough for stronger promotion.

### Typical signs
- multiple phrasings still compete
- the current wording still causes confusion
- the public label is not yet carrying the right boundary cleanly

---

## 2. Boundary patch-sensitive

### Meaning
The idea is useful, but its neighboring distinctions are still too unstable for stronger public status.

### Typical signs
- the branch bleeds into adjacent families or subtrees
- the boundary guide still cannot separate it cleanly
- too much of its identity depends on deleting ambiguity

---

## 3. Evidence patch-sensitive

### Meaning
The structure looks promising, but the public support base is still too thin for stronger status.

### Typical signs
- worked examples exist, but remain sparse
- validation is directional rather than stable
- observability weakness still affects the judgment

---

## 4. Repair patch-sensitive

### Meaning
The structure may help classification, but its downstream repair meaning is not yet stable enough.

### Typical signs
- first-fix direction still changes too much
- misrepair consequences are not yet well understood
- the branch has not yet earned stable action relevance

---

## 5. Rendering patch-sensitive

### Meaning
The structure may be real, but the public output/render discipline around it is still unstable.

### Typical signs
- people render it inconsistently
- the same idea appears with incompatible output shapes
- optional language keeps inflating beyond support

---

## 6. Overlay patch-sensitive

### Meaning
The structure may interact meaningfully with another surface, but the mixed-surface language is not yet stable enough for stronger public treatment.

### Typical signs
- secondary-family notes still blur into boundary uncertainty
- overlay wording is still too dependent on local interpretation
- the mixed-surface reading is not consistently useful

---

## Patch-sensitive summary table 📚

| Patch-sensitive class | What it means | Main reason to keep it patch-sensitive |
|---|---|---|
| Wording patch-sensitive | naming is still moving | language is not yet stable enough |
| Boundary patch-sensitive | separation is still unstable | neighboring structures are still too live |
| Evidence patch-sensitive | support is still thin | stronger promotion would outrun proof |
| Repair patch-sensitive | downstream action meaning is still moving | first-fix and misrepair consequences are not yet stable |
| Rendering patch-sensitive | output use is still inconsistent | public usage shape is not yet clean |
| Overlay patch-sensitive | mixed-surface expression is still unstable | secondary-family and overlay reading is not yet bounded enough |

## Threshold criteria for stronger promotion

A structure should usually move upward only when enough of the following are true.

### 1. Evidence quality
The structure has enough support to justify its public status.

Questions to ask:
- is the support direct enough
- is the support more than a one-off example
- is the support less dependent on rhetorical intuition

### 2. Boundary stability
The structure is distinguishable enough from neighboring structures.

Questions to ask:
- does it still collapse into nearby branches too easily
- can it be separated without deleting half the evidence
- does the naming improve clarity rather than blur it

### 3. Routing usefulness
The structure materially improves route-first reading.

Questions to ask:
- does it help classification
- does it reduce confusion
- does it improve interpretability

### 4. Repair usefulness
The structure has stable enough consequences for safer first action or safer restraint.

Questions to ask:
- does it reduce misrepair
- does it help first-fix discipline
- does it support safer next questions

### 5. Wording stability
The public language around the structure is clean enough to hold steady.

Questions to ask:
- does the wording still wobble
- do different readers mean different things by the same phrase
- can the label survive reuse without distorting itself

### 6. Output stability
The structure can be rendered consistently in public output.

Questions to ask:
- does the output contract handle it cleanly
- does it stay bounded in compact and expanded form
- does it avoid optional-field inflation

### 7. Cross-page consistency
The structure fits the rest of the Atlas stack cleanly.

Questions to ask:
- does it align with family, boundary, fit, evidence, and output layers
- does it require silent rewrites elsewhere to work
- does it create more contradictions than clarity

These criteria are why promotion should stay hard-earned.

---

## Promotion threshold table

| Threshold criterion | Low | Medium | High |
|---|---|---|---|
| Evidence quality | mostly suggestive | useful but still limited | strong enough for stable public use |
| Boundary stability | highly blurry | workable with caution | clean enough for dependable use |
| Routing usefulness | decorative | somewhat clarifying | materially clarifying |
| Repair usefulness | unclear | situational | consistently useful or safely constraining |
| Wording stability | still moving | mostly workable | stable across reuse |
| Output stability | inconsistent | usable with caution | stable in public rendering |
| Cross-page consistency | creates drift | partly aligned | cleanly aligned |

A structure generally should **not** be promoted on one strong dimension alone.

Upward movement should reflect enough stability across several dimensions.

## Anti-inflation rules

A good threshold page must also say what does **not** count.

### Rule 1. Repetition is not promotion
Just because a phrase appears many times does not mean it has earned stronger status.

### Rule 2. Popularity is not promotion
A useful or exciting concept is not automatically a stable structural unit.

### Rule 3. A page existing is not the same as the page being highly promoted
Public decomposition is not the same thing as frozen support.

### Rule 4. A good example is not a full branch
One clean worked case can teach well without justifying a new stable structure.

### Rule 5. More detail is not more truth
Sometimes detail is growth.  
Sometimes it is just noise wearing a lab coat.

### Rule 6. Patch activity is not core failure
Patch work often means healthy controlled growth, not instability of the mother structure.

These rules are extremely important.

Without them, Atlas could inflate itself simply by talking more.

---

## When something should stay lower-status

A structure should usually stay referenced only, teaching support, or patch-sensitive when:

- the evidence is still thin
- the boundary is still unstable
- the naming is still moving
- the routing value is still weak
- the repair consequences are still unclear
- the output use is inconsistent
- the concept is still more explanatory than structural

Staying lower-status is not failure.

It is often the most disciplined choice.

---

## When something may deserve stronger status

A structure may deserve upward movement when:

- the evidence is strong enough
- the boundary is clean enough
- the wording is stable enough
- the routing usefulness is real
- the repair consequences are clearer
- the output rendering remains controlled
- the structure aligns with the rest of the Atlas without hidden rewrites

Even then, promotion should remain explicit and traceable.

---

## Common promotion mistakes

### Mistake 1. Promoting because the wording sounds elegant
A beautiful phrase is not a structural threshold.

### Mistake 2. Promoting because the example feels persuasive
A compelling example is still only one slice of support.

### Mistake 3. Promoting because a page already exists
A page can exist as public decomposition without being a stronger class of structure.

### Mistake 4. Promoting because the pattern is emotionally satisfying
Satisfaction is not validation.

### Mistake 5. Confusing support-layer maturity with core maturity
A stable support page can stay support.

### Mistake 6. Treating patch sensitivity like embarrassment
Patch-sensitive status is often the correct honest status.

## How this page should affect Atlas growth

This page should change behavior in a simple way.

When a new page, branch, term, or pattern appears, Atlas growth should ask:

1. what class is it at now  
2. what criteria does it already meet  
3. what criteria does it still fail  
4. should it stay patch-sensitive  
5. what exact upward move, if any, has been earned

That process is boring in the best possible way.

It blocks ego-driven mutation.

## Practical use

Here is the simplest practical workflow.

### Step 1
Identify the thing being evaluated.
Is it:
- a wording
- a branch
- a page
- a teaching surface
- a mixed-surface pattern
- a support-layer asset

### Step 2
Assign the lowest honest current status.

### Step 3
Check the threshold criteria:
- evidence
- boundary
- routing usefulness
- repair usefulness
- wording stability
- output stability
- cross-page consistency

### Step 4
If multiple criteria are still weak, keep it patch-sensitive or lower-status.

### Step 5
Only promote upward when the structure has actually earned it.

That sequence keeps Atlas growth explicit.
